# Lessons Learned

> Append-only register of recurring rules and patterns. Re-read at start by /10x-frame, /10x-research, /10x-plan, /10x-plan-review, /10x-implement, /10x-impl-review.

## Read the design system before building any UI slice

- **Context**: Any user-facing slice (a UI screen or component), during both planning and implementation phases.
- **Problem**: Neither /10x-plan nor /10x-implement auto-discovers the design system, so UI gets built with no design reference — wrong tokens, spacing, structure, or copy.
- **Rule**: For any user-facing slice, read context/foundation/design-system.md first, then open only the matching screenshot(s) and source JSX for that slice. Build against the live tokens in src/styles/global.css; never import from context/foundation/design/ (it's a static prototype). Polish UI copy is canonical.
- **Applies to**: plan, implement

## Port the design spec — not the code — at exact fidelity, with a vision-diff gate

- **Context**: Any user-facing slice that maps to Claude Design mockups, across plan → implement → review — whenever a screen is built or changed against a design.
- **Problem**: The earlier version of this lesson optimized for _token cost_ — "distill the screenshot into a compact textual contract, then never re-open the PNG." That distillation is **lossy in the one dimension that defines layout fidelity**: it captured colors as exact hex but radii/spacing/structure as _ranges and prose_ ("cards 14–18px"), which read as implementer license. It also pushed vision out of the loop, so nothing compared the rendered app to the mockup until a human did it by hand — that's what a late "design-followups" punch-list is. Net: high freedom, low fidelity.
- **Rule — fidelity is the priority, not token cost**:
  1. **Port the spec, not the code.** Claude Design is code-backed (`DesignSync get_file` returns JSX). Transcribe the exact _values_: colors → our token names, radii, spacing, font size/weight, letter-spacing, uppercase-vs-sentence, breakpoint behavior, grid ratios, column order, what stacks on mobile, every component state, and the **verbatim Polish copy**. Tailwind class _values_ transfer directly (`rounded-[16px]`, `grid-cols-[1.35fr_1fr]`, `text-[11px] uppercase tracking-wide`) — apply them to _our_ tree. **Never copy their component code 1:1** — it isn't Astro/React-shaped; re-author the architecture in our idioms (Astro shell + React islands, `cn()`, shadcn, RHF for big forms).
  2. **Exact values, not ranges.** "14–18px" is a license to diverge — write the specific value per element. If a slice's or state's JSX isn't pulled, pull it: missing source = invention.
  3. **Mark each spec line `exact` vs `deviation(reason)`.** Deliberate divergences (StaffShell-standalone, single-column issue view, crimson CTA) must be recorded so the fidelity gate converges instead of re-flagging them.
  4. **Persist the mockups; the user provides them at plan time.** Screenshots live in the repo and stay available across plan → implement → review. On disk they cost nothing — only _loading a PNG into the main loop_ costs — so load them into **subagents** (extract the spec at plan; spot-check ambiguous slices at implement; diff at review). Two-tier retention: promote the canonical shipped-surface mockup to `context/foundation/design/screenshots/` (kept forever = fidelity baseline); prune per-change working shots from `<change>/design-review/` at archive. (Deleting later doesn't reclaim git space — the blob is permanent once committed.)
  5. **Close every UI slice with a vision-vs-mockup diff gate.** Render the real component at the mockup's breakpoints, hand the screenshot + the mockup PNG to a vision subagent, take back a structured punch-list, and iterate to empty (minus the marked deviations). This gate compares against the **mockup**, not a stored baseline of our own render — it is the thing that actually verifies fidelity.
- **Applies to**: plan, implement, impl-review

## Use react-hook-form for larger form islands

- **Context**: Any React (TSX) form island with many fields (roughly 8+), e.g. `src/components/fleet/VehicleForm.tsx` (~18 fields). Not small forms like `ReservationForm`.
- **Problem**: The plain `useState` + manual `fieldErrors` map pattern gets boilerplate-heavy and re-render-noisy as field count grows. A misconception that "react-hook-form isn't recommended with Astro" pushes people to keep hand-rolling state — but RHF runs fine inside an Astro island (it's just client React; the only nuance is island bundle size).
- **Rule**: For larger/complex form islands, prefer `react-hook-form` + `@hookform/resolvers` (`zodResolver(sharedSchema)`) over plain `useState`; keep the zod schema as the single validation source for client + API. Reserve plain `useState` for small forms. For non-interactive forms that don't need an island at all, the Astro-native choice is a plain `<form method="POST">` to an API route (works without JS).
- **This rule is forward-looking — apply it even though existing forms don't.** `VehicleForm.tsx` uses plain `useState` and predates nothing: it landed at `14db20a` (2026-06-26 15:39), **after** this lesson was written at `5d13d95` (2026-06-26 14:05), and simply did not adopt it. So `VehicleForm` is a **known exception, not a precedent** — do not cite it as evidence that "the codebase convention is `useState`", and do not let a "match the surrounding code" instinct override this rule. New large forms use RHF. First adopter: the S-05 issue-protocol form (`useFieldArray` for its dynamic photo rows). Migrating `VehicleForm` is a worthwhile follow-up, not a prerequisite.
- **Applies to**: plan, implement, impl-review

## API routes are outside middleware's gate — every /api route must self-gate

- **Context**: Any HTTP route under `src/pages/api/`, during both planning and implementation. Touches every current and future API endpoint (e.g. an admin-only `/api/invoices`).
- **Problem**: `src/middleware.ts` only enforces auth/role for paths listed in `ROUTE_ROLES` (`src/lib/access.ts:27-38`), which contains **only `/dashboard*` page prefixes** — no `/api/*` entry. So `resolveRequiredRole("/api/...")` returns `null` and middleware calls `next()` without any check. A new API route that forgets its in-handler gate is silently reachable by anonymous or wrong-role callers. Middleware _can't_ cleanly cover `/api` anyway: its unauthenticated branch is a 302 redirect to `/auth/signin` (page-shaped), not the 401/JSON an API client needs, and routes are non-uniform (`POST /api/reservations` is deliberately public; `vehicles` is staff-only). Writes have a DB backstop (RLS `WITH CHECK` / RPC `current_app_role()`), but reads via a definer RPC or service-role client do not.
- **Rule**: Treat every `/api` route as unprotected by default. Each handler must self-gate **in this order** before any work: (a) same-origin CSRF check on mutations (`origin !== context.url.origin` → 403), (b) auth (`!context.locals.user` → 401), (c) role (`!requireRole(context.locals, "<min>")` → 403), then (d) zod parse → 400, then the DB call. Use `json(status, …)` bodies, never a redirect. A route that is intentionally public must say so in a comment. Reference: `src/pages/api/vehicles.ts:30-63`. Do NOT rely on middleware or `ROUTE_ROLES` to protect API paths.
- **Applies to**: plan, implement, impl-review

## Wrap auth calls and role helpers in (select …) inside RLS policies

- **Context**: Any RLS policy `USING` / `WITH CHECK` clause that gates on the caller — `auth.uid()`, `auth.jwt()`, `current_setting(...)`, or a role helper like `public.current_app_role()`. Touches every authed slice (e.g. `supabase/migrations/20260604153139_employee_admin_roles.sql`, `20260625120000_fleet_management.sql`, and upcoming S-05/S-06/S-07/S-08).
- **Problem**: A bare `auth.<fn>()` / `current_setting()` call in a policy predicate is re-evaluated once per scanned row (O(rows)), which Supabase's `auth_rls_initplan` advisor flags as suboptimal at scale. A `STABLE SECURITY DEFINER` helper does NOT escape this — `STABLE` is necessary but not sufficient for one-time evaluation; the planner only hoists a _scalar subquery_ to a one-time InitPlan. The advisor only matches literal `auth.*()`/`current_setting()` tokens, so helper-based policies carry the same cost silently and go unflagged.
- **Rule**: Wrap every per-row caller check in a scalar subquery: `user_id = (select auth.uid())`, `(select public.current_app_role()) = 'admin'`. Wrap the role helper too, not just raw `auth.uid()`. Result is identical; evaluation drops to once per statement. Schema-qualify the helper (`public.current_app_role()`). This is a pure-perf change — fix it with `ALTER POLICY` in place (reversible by a symmetric `ALTER`); no type regen, no app-code change.
- **Applies to**: plan, implement, impl-review

## Polish text breaks every Latin-1 / WinAnsi output path — assume UTF-8 is not enough

- **Context**: Any code path that renders app text into a byte format with its own font or character encoding, rather than into a UTF-8 web page. Concretely: PDF generation (pdf-lib, jsPDF, PDFKit), CSV/XLSX exports opened in Excel, barcode/label printers, and any library whose default font is one of the 14 standard PDF fonts. First hit in S-05 (issue-protocol PDF); S-06 inherits it. This is a **Polish-only app** (PRD Non-Goals), so real data always contains these characters.
- **Problem**: The 14 standard PDF fonts are **WinAnsiEncoding (Windows-1252 / Latin-1)**, which covers Western European accents but **not Latin-2**. Of the nine Polish diacritics `ą ć ę ł ń ó ś ź ż`, **only `ó` exists in Windows-1252 — the other eight are absent**, and pdf-lib's `drawText` **throws** on any unencodable character (it does not silently substitute). The trap is that this is invisible in development: a prototype using `StandardFonts.Helvetica` with fixture data like `"Jan Kowalski"` / `"Test vehicle"` passes every test, and production 500s on the first customer named _Wąsik_ or a damage note reading _"zarysowanie lewego błotnika"_. Being "UTF-8 everywhere" in the database and the app does not help — the failure is at the **font/encoding boundary**, not the string.
- **Rule**: On any such path, never use a built-in/standard font. Embed a Unicode TTF and subset it: `pdfDoc.registerFontkit(fontkit)` + `pdfDoc.embedFont(ttfBytes, { subset: true })` (pdf-lib), then draw with that font. Budget the cost: `@pdf-lib/fontkit` ≈ +342 KB gzip, the source TTF ≈ 150–600 KB (Noto/DejaVu Sans), and ~10–40 KB of embedded subset per output file — all free static-asset weight if generation happens client-side. **Make the full diacritic set a test fixture, not an afterthought**: every fixture and every seeded name/note on such a path must contain `ą ć ę ł ń ó ś ź ż` (and `Ą Ć Ę Ł Ń Ó Ś Ź Ż`), so the encoding boundary is exercised by default. For CSV/Excel exports the sibling fix is a UTF-8 BOM; for anything else, verify the encoding explicitly rather than assuming.
- **Applies to**: plan, implement, impl-review

## Revoke EXECUTE before granting it — a grant alone never restricts an RPC

- **Context**: Any migration under `supabase/migrations/` that creates a function in the `public` schema — every `SECURITY DEFINER` RPC, which in this codebase is the sole write/read path across the RLS boundary. Touches S-02/S-03's existing RPCs and every future authed slice (S-06/S-07/S-08).
- **Problem**: Postgres grants EXECUTE on a new function to PUBLIC, and Supabase's `ALTER DEFAULT PRIVILEGES` adds explicit anon/authenticated/service_role grants on top. So `grant execute ... to authenticated` re-adds a role that is already there — it restricts NOTHING. Verified 2026-07-10: all four staff RPCs were anon-callable; each refused only because of its in-RPC `current_app_role()` gate (no leak, no state change). The rule's value is prospective: one definer RPC that forgets that hand-written gate is silently callable by anyone with the public anon key. Same default-grant family as the confirmed `reservations` PII leak — but a table has no in-function guard, which is why that one WAS exploitable.
- **Rule**: Never rely on `grant execute ... to authenticated` to restrict an RPC — it is a no-op against the default grants. Always `revoke execute on function ... from public, anon;` FIRST, then grant to the roles that need it — **per function, every time**. There is NO reliable schema-level "start closed" default to lean on: `alter default privileges in schema public revoke execute on functions from public, anon, authenticated;` was spike-verified a **no-op in this Supabase environment** (2026-07-14, change `rpc-execute-grant-hardening`) — a freshly-created function still gets the built-in PUBLIC execute grant (`proacl {=X/...}`), reproduced three ways. So the per-function revoke is the durable control; do it on every new RPC. Two carve-outs: (a) an intentionally-public RPC must carry an explicit `grant execute ... to anon` and say so in a comment; (b) a helper invoked from inside an RLS policy (e.g. `current_app_role()`) executes as the QUERYING role and needs its own explicit grant to `authenticated`, or every policy referencing it errors. Keep the in-RPC `current_app_role()` gate regardless — defense in depth, not either/or.
- **Applies to**: plan, implement, impl-review

## Buttons keep the default cursor (Tailwind v4 / shadcn) — restoring pointer risks a hover flicker

- **Context**: shadcn/ui buttons on Tailwind v4 — the shared `Button` (real `<button>` + `asChild` link-buttons), across every UI slice.
- **Problem**: Tailwind v4 + shadcn default buttons to `cursor: default` (shadcn's Feb-2025 v4 note: "buttons now use the default cursor"). It's tempting to "restore" `cursor: pointer` since users expect it — but doing so surfaced a **pointer↔default flicker while moving the mouse over buttons**: global, reproducible in Incognito on healthy hardware (GPU confirmed fine via `chrome://gpu`), and `transition-all`→`transition-colors` did NOT fix it (2026-07-21). Anchor-buttons (`asChild` → `<a>`) also carry the UA pointer, so a partial fix flickers too.
- **Rule (project decision 2026-07-21)**: Buttons use the **default cursor** — do NOT add `cursor-pointer`; keep `button.tsx` at shadcn stock. To force the default consistently across real `<button>`s AND `asChild` links (which otherwise inherit the UA anchor pointer), use one base-layer rule on the shared slot in `src/styles/global.css`:
  ```css
  @layer base {
    [data-slot="button"] {
      cursor: default;
    }
  }
  ```
  If a future project genuinely wants pointer on buttons, know it's a UX toss-up and can flicker under some compositing paths — verify the hover-move behaviour before committing to it.
- **Applies to**: implement, impl-review

## Locale/timezone/currency are single-locale (`pl-PL` / `Europe/Warsaw`) — pin the zone on SSR, centralize before multi-country

- **Context**: Any code formatting a date/number/currency or rendering user-facing copy. Today every site hardcodes `pl-PL` / `Europe/Warsaw` inline; there is no i18n layer.
- **Problem**: (a) An SSR'd component (`client:load`/`client:visible`) formatting a time with a runtime-relative formatter (no explicit `timeZone`) renders in the **server's** zone (workerd = UTC) then re-renders in the **browser's** zone → React **hydration mismatch** (seen 2026-07-22 on the return signature timestamp: server 10:49 vs client 12:49). (b) More broadly, the app is single-locale by construction, so any non-PL deployment is a wall.
- **Rule**: For a new timestamp/number/currency display: (1) **always pass an explicit `timeZone` when the value is SSR'd** — never rely on the runtime zone. (2) A **company-anchored** event time (signature, pickup, return) is shown in the **company zone** (`Europe/Warsaw`), NOT the viewer's — viewer-local is only for personal/relative times ("your session expires…"). Viewer-local would also reintroduce the SSR mismatch (the server can't know the viewer's zone). (3) Prefer a single named constant/helper over an inline `pl-PL`/`Europe/Warsaw` string so the eventual i18n seam is one place. Do NOT build full i18n speculatively (YAGNI) — tracked in `known-issues.md` → "Single-locale by construction". Mind **workerd's trimmed ICU** for server-side locale formatting (why `returns.astro` hand-rolls Polish month names).
- **Applies to**: plan, implement, impl-review

## End every UI-touching plan with a Design Alignment Audit gate

- **Context**: `/10x-plan` for any slice that creates or edits a user-facing surface (`src/pages/**` non-API, `src/components/**`, `src/layouts/**`, `src/styles/**`). Skip for backend / data / API / test-only plans.
- **Problem**: Design fidelity was governed only by convention — the two design lessons above plus an ad-hoc "Phase 1" someone remembered to author (S-05, S-06). Nothing in `/10x-plan` forced a check that the repo's designs were still current, that newly provided mockups were complete, or that every UI phase actually maps to a design. So a plan could be declared "done" while pointing at a **stale** screenshot (e.g. the catalogued overdue-returns mockups `18`/`24`, which predate the S-07 redesign) or while a UI surface had no design at all — the gap only surfacing mid-implementation. `/10x-plan` had zero mention of design/screenshots/fidelity anywhere in its body (verified 2026-07-23).
- **Rule**: For a UI-touching plan, planning is not complete until a terminal **Design Alignment Audit** passes (`/10x-plan` Step 6). It is a _paper_ audit at plan time — the **rendered** vision-diff stays in implement/impl-review (the "Port the design spec …" lesson). It (1) captures the user's final canonical screenshots into `context/changes/<id>/design-review/`; (2) freshness-audits repo designs (`design-system.md` rows + screenshots) as current / superseded / missing; (3) quality-audits the new designs for missing states / breakpoints / copy; (4) writes/refreshes `design-contract.md` with **exact values (never ranges)**, verbatim Polish copy, each line `exact` / `deviation(reason)`; (5) confirms every canonical surface has a plan phase and vice-versa, looping back to revise the plan on any mismatch. Canonical screenshots are provided by the user as the last step of planning; without them the session cannot close.
- **Generic skills, project-pinned specifics**: `/10x-plan` (Step 6), `/10x-implement`, and `/10x-impl-review` state the exact-values / no-ranges / design-audit rule **generically** — no language or framework is baked into them. This project pins the specifics _here and in the design docs_: a "user-facing surface" is `src/pages/**` (non-API), `src/components/**`, `src/layouts/**`, `src/styles/**`; UI copy is **`pl-PL`, canonical** (see `design-system.md` "Polish-first copy" + the "Polish text breaks every Latin-1 / WinAnsi output path" lesson); design tokens live in `src/styles/global.css`; the design source is the Claude Design project named in `design-system.md`. When applying these skills in another project, override those pins in that project's rules — do not edit the skills.
- **Applies to**: plan (the gate); implement, impl-review (consume the contract + run the rendered vision-diff)

## Embeddable panels: size internal layout by container width, not the viewport

- **Context**: Any component that can render in more than one width context — a master-detail detail pane, or a panel extracted for reuse (e.g. `RequestDetail` in `src/components/dashboard/PendingQueue.tsx`, extracted "so the calendar can reuse it"). NOT full-page forms/views (`ProtocolForm`, `ReturnProtocolForm`, `VehicleDetail`, `ReservationForm`), which own the page width and where `md:`/`lg:` splits are safe.
- **Problem**: `md:`/`lg:` breakpoints measure the browser **window**; an embedded panel's real width is the **column it sits in**. `RequestDetail`'s vehicle/dates split was `md:grid-cols-2`, but inside the `md:grid-cols-[360px_1fr]` master-detail the detail column was ~298px even at a 954px viewport, so the 2-up ran at ~143px/card and the fixed 96px truck icon clipped the text (2026-07-22). Insidious because it stays green on every gate — types, lint, unit, integration, even a naive E2E `toBeVisible()` — since the nodes exist and the viewport reads as desktop; it only surfaces at an in-between width in the embedded context. The sweep confirmed reservations is the app's **only** inline master-detail (pickups/returns route to full-page `[reservationId]` views, so their `lg:grid-cols-[1.35fr_1fr]` splits are page-level and safe).
- **Rule**: If a component can render inside a variable-width container, drive its internal layout from its own width with Tailwind v4 container queries — mark the panel `@container` and use `@min-[Npx]:` (e.g. `@min-[520px]:grid-cols-2`; never the bare `@[520px]:`, which emits no CSS). Reserve `md:`/`lg:` for components guaranteed to own the full page. When a master-detail collapses, raise the split to where the detail has room (here `md`→`lg`) and move the panel's back-button header to the same boundary so the narrow view stays navigable. Verify by rendering at in-between + embedded widths, not only the mockup's canonical breakpoints — extends the "Port the design spec … with a vision-diff gate" lesson.
- **Applies to**: plan, implement, impl-review
